Day 10 — 本番インフラ整備（PostgreSQL移行・Puma/Sidekiq強化）

12. 概要

本番環境のインフラ基盤を整備した。SQLiteからPostgreSQLへの移行準備、Pumaのクラスターモード化、Sidekiqの本番Redis接続設定、デプロイ基盤（Procfile・リリーススクリプト）を構築。

コンポーネント修正前修正後
データベースSQLite3（開発/本番共通）開発: SQLite3 / 本番: PostgreSQL
WebサーバーPuma 3.x（シングルプロセス）Puma 5.x（本番: クラスターモード）
ジョブキュー:async（メモリ内、再起動で消失）:sidekiq（Redis永続化）
デプロイ手動Procfile + bin/release

13. SQLite → PostgreSQL 移行

Gemfile変更

# 修正前 gem 'sqlite3', '~> 1.6' # 修正後 gem 'pg', '~> 1.5', group: :production gem 'sqlite3', '~> 1.6', groups: [:development, :test]

database.yml変更

環境アダプター接続先
developmentsqlite3db/development.sqlite3（変更なし）
testsqlite3db/test.sqlite3（変更なし）
productionpostgresqlDATABASE_URL 環境変数（優先）または個別設定

本番PostgreSQL接続設定

production: adapter: postgresql encoding: unicode pool: <%= ENV.fetch("RAILS_MAX_THREADS") { 15 } %> url: <%= ENV['DATABASE_URL'] %> host: <%= ENV.fetch('DB_HOST', 'localhost') %> database: <%= ENV.fetch('DB_NAME', 'ai_interviewer_production') %> username: <%= ENV.fetch('DB_USERNAME', 'ai_interviewer') %> password: <%= ENV['DB_PASSWORD'] %>
移行手順: 開発環境はSQLiteのままなので、既存のワークフローに影響なし。本番デプロイ時にPostgreSQLを使用する。DATABASE_URL が設定されている場合（Render/Heroku等）、個別設定よりも優先される。

14. Puma 本番設定強化

ファイル: config/puma.rb

設定修正前修正後
バージョンPuma 3.xPuma 5.x
ワーカー数未設定（シングルプロセス）本番: WEB_CONCURRENCY（デフォルト2）
preload_app無効本番: 有効（メモリ効率化）
DB再接続未設定on_worker_boot で establish_connection
スレッド最小=最大=5RAILS_MIN_THREADS / RAILS_MAX_THREADS で個別設定可
注意: Puma 3.x → 5.x のアップグレードにより、bundle update puma が必要。互換性のある変更だが、本番デプロイ前にステージング環境で動作確認を推奨。

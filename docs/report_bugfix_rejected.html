2. 原因分析

2.1 修正前コード

ファイル: app/models/interview_result.rb

def rejected?
  rejection_details.present? && rejection_details != {}
end

2.2 根本原因：SQLite の JSON カラム型の挙動

DBスキーマでは rejection_details カラムは以下のように定義されています:

t.json "rejection_details", default: {}

ここで問題となるのが SQLite の JSON カラムの内部表現 です。

【PostgreSQL（本番環境）の場合】 DB格納値: {} (ネイティブ JSON型) Rails読取: {} (Ruby Hash) {}.present? → false ✅ 正常動作 【SQLite（開発/テスト環境）の場合】 DB格納値: "{}" (TEXT型として格納) Rails読取: "{}" (Ruby String) "{}".present? → true ❌ 意図しない動作
原因まとめ: SQLite は JSON 型をネイティブサポートしておらず、TEXT 型として格納します。 Rails が値を読み取ると Ruby の String "{}" として返されます。 String#present? は空文字でない限り true を返すため、 空ハッシュであるはずの rejection_details が「存在する」と誤判定されました。

2.3 比較表

チェック式PostgreSQL ({} Hash)SQLite ("{}" String)
.present?false ✅true ❌
!= {}false ✅true ❌
.blank?true ✅false ❌

つまり修正前のコードは PostgreSQL 前提 であり、SQLite 環境では常に true を返してしまう問題がありました。

3. 修正内容

3.1 修正後コード

ファイル: app/models/interview_result.rb:35-40

def rejected?
  details = rejection_details
  return false if details.blank?

  parsed = details.is_a?(String) ? (JSON.parse(details) rescue nil) : details
  parsed.is_a?(Hash) && parsed.any?
end

3.2 修正ロジックの説明

ステップ処理内容目的
1details.blank? チェックnil・空文字・空ハッシュを早期リターン
2is_a?(String) 判定SQLite が返す文字列型を検出
3JSON.parse + rescue文字列を安全にHashへ変換
4parsed.is_a?(Hash) && parsed.any?中身のあるHashの場合のみ true

5. 影響範囲と今後の注意事項

5.1 影響範囲

対象影響
本番環境 (PostgreSQL)影響なし — PostgreSQL はネイティブ JSON 型を使用するため、修正前でも正常動作
開発/テスト環境 (SQLite)修正により正常動作 — 文字列型の JSON 値を正しく処理
面接結果の不合格判定SQLite 環境で「全結果が不合格」と誤表示されていた可能性を解消

5.2 今後の注意事項

  1. DB間の型差異に注意: SQLite と PostgreSQL で json カラムの挙動が異なるため、JSON カラムを扱うメソッドでは型チェックを行うこと
  2. 開発/テスト環境の統一検討: 本番と同じ PostgreSQL を開発環境でも使用することで、この種の不整合を根本的に防止可能
  3. 類似パターンの確認: 他のモデルで JSON カラムのデフォルト値 {} を present? で判定している箇所がないか定期的にチェック
